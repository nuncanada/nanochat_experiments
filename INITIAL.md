This transformation should be isomorphic at first, storing that layer 2 depends on layer 1, layer 3 depends on layer 2 blocks and so forth.


And then we want to reutilize/recreate as much as possible the code in https://github.com/karpathy/nanochat to create a process to retrain this new language model to flexibilize as much as possible these new dependence parameters to allow inference to be run faster with less steps depending on the number of layers.

### 1. New Model Architecture: `RecursiveGPT`
I created a new class `RecursiveGPT` in `nanochat/recursive.py`. This model inherits from the standard `GPT` but introduces a **dependency matrix** $D$ (initially an identity matrix) that defines which blocks are executed at each recursive step.
- **Isomorphism:** When $D=I$, the model behaves exactly like the original 12-layer Transformer.
- **Flexibility:** The dependency matrix is a learnable parameter, allowing the model to learn skip-connections, parallel block execution, or even reordering of layers.
- **Recursive Logic:** The `forward` method iterates through $T$ steps, and in each step, it applies a weighted combination of block updates.

### 2. Transformation Script: `scripts/transform_to_recursive.py`
This script converts a standard `GPT` checkpoint into a `RecursiveGPT` checkpoint. It:
- Loads the original weights.
- Initializes the `dependency_matrix` as an identity matrix to ensure the model remains isomorphic to the original.
- Saves the new checkpoint in a format compatible with the recursive trainer.

### 3. Retraining Process: `scripts/recursive_train.py`
This script allows retraining the recursive model to "flexibilize" the dependency parameters.
- **Speed-up:** You can specify `--n-steps < n_layer` to train a model that achieves similar results in fewer sequential steps, effectively running blocks in parallel or skipping redundant ones.
- **Monitoring:** The script periodically prints a visual representation of the dependency matrix to show how the "graph" of dependencies is evolving.
- **Muon Optimization:** It uses the same advanced optimizer structure as `nanochat`, including Muon for matrix parameters and AdamW for the new dependency parameters.

## MODEL ✨

The speedup for inference depends entirely on the **sparsity** and **rank** of the learned dependency matrix $D$ after retraining. However, we can estimate the theoretical upper bounds and practical bottlenecks:

### 1. Theoretical Speedup: $O(N_{layers} / N_{steps})$
If you successfully retrain a 12-layer model to work in 6 recursive steps without losing significant accuracy, the theoretical speedup is **2.0x** for the transformer trunk.
*   **Sequential Bottleneck:** In a standard Transformer, layer $n+1$ cannot start until layer $n$ finishes. 
*   **Recursive Parallelism:** In the `RecursiveGPT` implementation, if the dependency matrix learns to execute multiple blocks in the same step $t$ (e.g., $D_{t, i} > 0$ and $D_{t, j} > 0$), those blocks can technically be computed in parallel.

### 2. Practical Hardware Speedup: 1.2x - 1.5x
While the "math" suggests a 2x speedup for 6 steps, several real-world factors will eat into that:
*   **Memory Bandwidth:** LLM inference is often memory-bound. Even if you compute blocks in parallel, you still need to load the weights for all 12 blocks from VRAM. If $N_{steps}$ is small, you are loading more weights per "step," which increases throughput but doesn't reduce the total data moved.
*   **Kernel Overhead:** Launching 12 small kernels (sequential) vs. 1 large fused kernel (parallel) is faster, but `RecursiveGPT` currently loops in Python/Torch. To see the full speedup, you would need to use `torch.compile` (which I included in the training script) to fuse the parallel block updates into a single GPU kernel per step.
*   **Non-Transformer Costs:** The Embedding layer, LM Head, and KV-cache management remain constant. If the transformer trunk is 80% of the inference time, a 2x trunk speedup results in a $1 / (0.2 + 0.8/2) = 1.66x$ total speedup.

### 3. The "Sweet Spot" Guess
Based on the **Tiny Recursive Model (TRM)** paper's results (which uses high recursion for reasoning but fewer physical layers for throughput):
*   **For Reasoning:** Inference might actually be **slower** if you increase $N_{steps}$ beyond $N_{layers}$ to allow the model to "think" longer (iterative refinement).
*   **For Standard Generation:** I would estimate a **30% to 50% (1.3x - 1.5x) speedup** is realistic if you can compress the "logic" of a 12-layer model into 8 highly parallelized recursive steps.

### Summary Table (Estimated)
| $N_{steps}$ | Efficiency Strategy | Estimated Speedup | Risk |
| :--- | :--- | :--- | :--- |
| **12** | Isomorphic (Base) | 0.95x (extra matrix mult overhead) | None |

**The real win** isn't just raw speed, but the ability to **dynamically trade time for accuracy**: you could theoretically use $N_{steps}=4$ for "easy" tokens (commas, "the") and $N_{steps}=16$ for "hard" tokens (logic, math) within the same model.
